# 完成前验证 — 证据先于断言

## 铁律

```
没有新鲜的验证证据，不得声称完成
```

如果你没有**在本次消息中**运行过验证命令，你不能声称它通过了。

## 门控函数

```
在声称任何状态或表达满意之前：

1. 识别: 什么命令能证明这个声明？
2. 运行: 执行完整命令（新鲜的、完整的）
3. 阅读: 完整输出，检查退出码，计数失败
4. 验证: 输出是否确认了声明？
   - 否: 报告实际状态和证据
   - 是: 带证据发表声明
5. 然后才: 发表声明

跳过任何一步 = 撒谎，不是验证
```

## 常见声明的验证要求

| 声明 | 需要 | 不够 |
|------|------|------|
| 测试通过 | 测试命令输出: 0 失败 | 上次运行结果、"应该通过" |
| Lint 干净 | Linter 输出: 0 errors | 部分检查、推断 |
| 构建成功 | Build 命令: exit 0 | Lint 通过不等于编译通过 |
| Bug 修复了 | 测试原始症状: 通过 | 代码改了就假设修好了 |
| 回归测试有效 | Red-Green 循环验证 | 测试只通过一次 |
| 需求满足 | 逐条对照检查表 | "测试过了" |

## Red Flags — 停下来

- 使用"应该"、"大概"、"似乎"
- 在验证前表达满意（"好了！"、"完美！"、"搞定！"）
- 准备 commit/push/PR 但没验证
- 信任子代理的成功报告（必须独立验证）
- 依赖部分验证
- 想着"就这一次"
- 累了想早点收工
- **任何暗示成功但没运行验证的措辞**

## 反合理化

| 借口 | 现实 |
|------|------|
| "现在应该好了" | 运行验证命令 |
| "我很有信心" | 信心 ≠ 证据 |
| "就这一次" | 没有例外 |
| "Lint 过了" | Lint ≠ 编译 ≠ 测试 |
| "子代理说成功了" | 独立验证 |
| "我累了" | 疲惫 ≠ 借口 |
| "部分检查够了" | 部分什么都证明不了 |

## 关键模式

**测试：**
```
✅ [运行测试命令] [看到: 34/34 pass] "所有测试通过"
❌ "现在应该通过了" / "看起来对的"
```

**构建：**
```
✅ [运行构建] [看到: exit 0] "构建通过"
❌ "Lint 过了"（Lint 不检查编译）
```

**需求：**
```
✅ 重读计划 → 创建检查表 → 逐条验证 → 报告差距或完成
❌ "测试过了，阶段完成"
```

**子代理委托：**
```
✅ 子代理报告成功 → 检查 VCS diff → 验证变更 → 报告实际状态
❌ 信任子代理报告
```

## 何时适用

**总是在以下之前：**
- 任何成功/完成声明的变体
- 任何满意表达
- 任何关于工作状态的正面声明
- Commit、PR 创建、任务完成
- 移到下一个任务

## 底线

**验证没有捷径。**

运行命令。读输出。**然后**声称结果。

不可协商。
