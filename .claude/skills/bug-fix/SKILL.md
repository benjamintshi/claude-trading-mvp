# Bug 修复流程

## 铁律

```
没有根因调查，不得提出修复方案
```

如果你还没完成 Step 2（分析），你不能跳到 Step 3（修复）。

**违反规则的字面意思就是违反规则的精神。**

## 流程 (5 步)

### Step 1: 报告
创建 bug 记录：

```markdown
# specs/bugs/BUG-{NNN}-{slug}.md

## 状态: open | analyzing | fixing | verifying | pivoting | closed

## 描述
[什么坏了]

## 复现步骤
1. ...
2. ...
3. 预期: ... 实际: ...

## 环境
- 版本/分支: ...
- 浏览器/OS: ...

## 严重程度: critical | major | minor
```

### Step 2: 分析
- 读取相关代码和日志
- 定位根因（不是症状）
- 记录分析结果到 bug 文件：
  ```markdown
  ## 根因分析
  - 根因: [具体代码位置和逻辑错误]
  - 影响范围: [哪些功能受影响]
  - 关联: [是否和其他 bug 相关]
  ```

### Step 3: 修复
- 实现最小修复
- **先写回归测试**（能复现 bug 的测试）
- 再修复代码让测试通过
- 确认不影响其他功能

### Step 4: 验证
- 回归测试通过
- 全量测试通过
- 更新 bug 文件状态为 `closed`
- 记录修复内容：
  ```markdown
  ## 修复
  - 修改文件: [list]
  - 修复方式: [描述]
  - 回归测试: test_xxx
  - 关闭日期: YYYY-MM-DD
  ```

### Step 5: Pivot 脱困（当 Step 3 反复失败时）

**触发条件**：同一个修复方向尝试 3 次仍失败（测试不过、新 bug 产生、编译错误增加）。

当陷入失败循环时，**停下来，不要继续同一方向**。执行 Pivot 流程：

#### 5a. 命名失败
```markdown
## Pivot 记录
### 尝试过的方向
1. [方向 A]: [做了什么] → [为什么失败]
2. [方向 B]: [做了什么] → [为什么失败]
3. [方向 C]: [做了什么] → [为什么失败]

### 失败模式
- [共同点: 这些尝试都假设了 X，但也许 X 是错的]
```

#### 5b. 生成 3 个真正不同的替代方案
使用以下思维工具，每个方案必须**本质不同**（不是同一思路的变体）：

| 思维工具 | 方法 | 示例 |
|----------|------|------|
| **反转** | 如果问题的反面是什么？ | "不修复这个函数，而是绕过它" |
| **类比** | 其他系统怎么解决类似问题？ | "像断路器模式一样处理失败" |
| **激进简化** | 如果只有 10 行代码的预算？ | "删掉整个功能重写比修补更快" |

```markdown
### 替代方案
- 方案 X (反转): [描述]
- 方案 Y (类比): [描述]
- 方案 Z (简化): [描述]
```

#### 5c. 快速验证
对每个方案做最小测试（5 分钟上限）：
- 能编译吗？
- 核心假设成立吗？
- 有没有立即的反面证据？

#### 5d. 选择并执行
选择信号最强的方案，回到 Step 3。

#### 5e. 升级
如果 Pivot 后再失败（共经历 2 轮完整 Pivot），升级给用户：
```
已尝试 {N} 个不同方向，均未成功:
1. [方向 + 失败原因]
2. [方向 + 失败原因]
...
建议: [需要更多信息 / 需要架构变更 / 需要人类判断]
```

## 常见合理化

| 借口 | 现实 |
|------|------|
| "问题很简单，不需要流程" | 简单问题也有根因。流程对简单 bug 很快。 |
| "紧急，没时间走流程" | 系统化调试比猜测试错**更快**。 |
| "先试一下，再调查" | 第一个修复设定模式。从一开始就做对。 |
| "先修好再补测试" | 没测试的修复不持久。先写测试。 |
| "同时改多处节省时间" | 无法隔离什么起了作用。还会引入新 bug。 |
| "我看到问题了，直接修" | 看到症状 ≠ 理解根因。 |
| "再试一次修复"（已失败 2+ 次） | 3+ 次失败 = 架构问题。质疑模式，不要再修。 |

## Red Flags — 停下来，回到 Step 2

如果你发现自己在想：
- "快速修一下，之后再调查"
- "试试改 X 看行不行"
- "同时改多处，跑测试"
- "跳过测试，手动验证"
- "大概是 X，修一下"
- "不完全理解，但可能管用"
- 列出解决方案但没做过调查

**以上全部意味着：停下来。回到 Step 2 分析。**

## 关键原则
- **先复现再修** — 不能复现的 bug 不要盲修
- **先测试再改代码** — 回归测试证明 bug 存在
- **修根因不修症状** — 不要打补丁
- **3 次失败就 Pivot** — 不在死路上浪费时间
- **记录留痕** — bug 文件和 pivot 记录是未来的参考
